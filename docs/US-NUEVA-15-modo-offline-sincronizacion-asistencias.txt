US-NUEVA-15: Modo Offline y Sincronizacion Automatica de Asistencias
Proyecto: MarcaYA
Responsable: Busti
Fecha: 2026-06-14

1. Objetivo de la historia

Implementar un flujo de marcacion de asistencia tolerante a perdida de conexion.
Cuando la app Flutter no tenga internet, debe guardar localmente la marcacion
con su hora real y estado PENDIENTE_SINCRONIZACION. Al recuperar conexion, la
app debe sincronizar las marcaciones pendientes en lote contra el backend Rails.
El backend debe guardar la hora original de la marcacion fisica, no la hora en
la que se ejecuto la sincronizacion.

3. Arquitectura implementada

Frontend Flutter:
- Se agrego deteccion de conectividad con connectivity_plus.
- Se agrego almacenamiento local con SQLite usando sqflite.
- Se agrego un modelo local para marcaciones pendientes.
- Se agrego un repositorio local para leer, guardar, actualizar y eliminar la
  cola offline.
- Se agrego un provider global para:
  - detectar conexion;
  - guardar marcaciones offline;
  - sincronizar automaticamente al recuperar conexion;
  - exponer contador de pendientes y mensajes de error;
  - limpiar la cola local solo cuando el backend confirma exito o duplicado.

Backend Rails:
- Se agrego un endpoint de sincronizacion en lote.
- Se agrego un caso de uso especifico para sincronizar marcaciones offline.
- Se extendieron los casos de uso existentes de entrada y salida para aceptar
  fecha_hora opcional y cliente_marcacion_id.
- Se agrego idempotencia mediante cliente_marcacion_id.
- Se agrego una migracion para guardar cliente_marcacion_id en
  registro_asistencias.

8. Endpoint nuevo

POST /api/v1/asistencia/sincronizar

Autenticacion:
- Requiere usuario autenticado.
- Requiere rol empleado.
- Usa el empleado asociado al usuario autenticado.

Payload esperado:

{
  "marcaciones": [
    {
      "cliente_marcacion_id": "offline-123456-10-ENTRADA",
      "parada_id": 10,
      "tipo_marcacion": "ENTRADA",
      "latitud": -12.119,
      "longitud": -77.034,
      "fecha_hora_original": "2026-06-14T13:20:00Z"
    }
  ]
}

Respuesta:

{
  "sincronizados": [],
  "duplicados": [],
  "fallidos": []
}

Codigos HTTP:
- 200: todas las marcaciones fueron sincronizadas o ya eran duplicadas.
- 207: hubo sincronizacion parcial con fallidos.
- 422: error general de validacion.
- 404: usuario empleado no asociado.

13. Preservacion de la hora original

Antes:
- marcar_entrada.rb y marcar_salida.rb usaban Time.now en el backend.

Ahora:
- Los casos de uso aceptan fecha_hora opcional.
- El endpoint de sincronizacion manda fecha_hora_original al caso de uso.
- El registro se guarda con esa hora original en registro_asistencias.fecha_hora.

Esto cumple el criterio:
"guardar la hora original en que se efectuo la marcacion fisica, no la hora de
sincronizacion".


14. Manejo de duplicados

Cada marcacion offline genera un cliente_marcacion_id.

Ejemplo:
offline-1780000000000000-10-ENTRADA

El backend revisa si ya existe un registro con ese identificador:
- si existe, no crea otro registro;
- lo devuelve dentro de duplicados;
- Flutter elimina esa marcacion local porque ya fue procesada anteriormente.

18. Riesgos tecnicos y soluciones

Riesgo:
connectivity_plus indica red disponible, pero el backend no responde.

Solucion:
Si falla la sincronizacion, la marcacion permanece en SQLite y se reintenta mas
adelante.

Riesgo:
Marcaciones duplicadas por reintento del mismo lote.

Solucion:
cliente_marcacion_id tiene indice unico y el caso de uso detecta duplicados.

Riesgo:
La app se cierra antes de sincronizar.

Solucion:
La cola SQLite persiste en disco y se vuelve a leer al iniciar la app.

Riesgo:
El usuario inicia la app online pero sin token cargado.

Solucion:
Ademas de sincronizar al recuperar conexion, se dispara sincronizacion despues
del login.

Riesgo:
Backend guarda hora de sincronizacion.

Solucion:
El endpoint de lote usa fecha_hora_original y los casos de uso aceptan fecha_hora
explicita.

Riesgo:
Datos incompletos o invalidos.

Solucion:
El backend responde esa marcacion dentro de fallidos y Flutter no la elimina de
la cola local.


19. Informe tecnico para Sebastian

## 1. Arquitectura Utilizada

Se implemento la sincronizacion offline respetando la separacion por capas.
En Flutter se crearon un modelo local, un repositorio SQLite, un servicio de
conectividad y un provider global. La pantalla de marcacion consume el provider
sin acoplarse a SQLite ni a la deteccion de red. En Rails se agrego un caso de
uso de sincronizacion en lote conectado desde AsistenciaFacade y expuesto por
AsistenciasController. La entidad RegistroAsistencia, mapper, repositorio y
serializer se extendieron para soportar cliente_marcacion_id.

## 2. Conceptos Implementados

La app detecta conectividad con connectivity_plus. Si no hay conexion, guarda la
marcacion en SQLite con estado PENDIENTE_SINCRONIZACION, coordenadas, tipo,
parada y hora real. Al recuperar conexion, AsistenciaOfflineProvider envia el
lote al backend. El backend valida cada marcacion, evita duplicados por
cliente_marcacion_id y guarda fecha_hora_original como fecha_hora real. Flutter
limpia la base local solo para registros sincronizados o duplicados confirmados.

## 3. Fallos Comunes y Solucion

Perdida de conexion durante sincronizacion:
La cola local no se elimina y se reintenta automaticamente.

Marcaciones duplicadas:
Se usa cliente_marcacion_id con indice unico y respuesta duplicados.

Error al limpiar almacenamiento local:
Solo se limpia despues de respuesta positiva; si falla, queda pendiente.

Backend guarda hora de sincronizacion:
Se evita usando fecha_hora_original enviada por Flutter.

Conflictos por datos incompletos:
El backend devuelve el item en fallidos y Flutter conserva la marcacion.

Fallos de permisos o ubicacion:
No se modifico la logica GPS existente; si no hay ubicacion, la pantalla no
permite confirmar.
